# MANDATORY SEPARATION OF INIT AND PROCESS — NON-NEGOTIABLE

This is the highest-priority structural rule. It overrides all other organizational decisions.

The initialization function and the main biophysical process function MUST always be two separate, independent Python functions. This is true regardless of how the original code is organized.

## What each function must contain

**Initialization function** — sets up initial state only:
- Computes and returns the initial values of all biophysical state variables.
- Contains only logic that runs once at the start of a simulation: default assignments, initial condition computations, parameter-derived setup values.
- Must NOT contain any time-step logic, rate computations, or per-step state updates.
- Must NOT call the main process function.

**Main biophysical process function** — advances state by one time step only:
- Receives current state variables as explicit input parameters.
- Computes rates, updates, and new state values for the current time step.
- Returns all updated state variables explicitly.
- Must NOT re-initialize state. Must NOT contain initialization logic.
- Must NOT call the initialization function.

## How to separate them when the original code mixes them

If the original code places initialization and per-step logic in the same block or function, you MUST refactor them into two separate functions. Follow this procedure:

1. Identify all statements that run only once (initial assignments, parameter-derived defaults, one-time computations). These belong in the initialization function.
2. Identify all statements that update state each time step (rate calculations, balance equations, conditional updates based on current state). These belong in the main process function.
3. Ensure no variable computed in the initialization function is recomputed in the main process function unless it is passed in as an explicit parameter.
4. Ensure the two functions share no mutable state, no globals, and no implicit dependencies. All data flows through explicit parameters and return values.
5. Verify that calling init once followed by calling process repeatedly produces numerically identical results to the original mixed code.

## Absolute prohibitions
- NEVER merge initialization and process logic into a single function.
- NEVER place a time-step update inside the initialization function.
- NEVER place an initial state assignment inside the process function unless that variable is explicitly passed in as a parameter.
- NEVER have the initialization function call the process function or vice versa.
- NEVER produce a module where these two functions are absent, merged, or where one wraps the other.

# HOW TO IDENTIFY SECTIONS

**Initialization function** — look for:
- Names like init, initialize, reset, setup, constructor, or clear patterns in docstrings/comments.
- Statements that assign fixed starting values to state variables (e.g. SnowWaterContent = 0.0).
- Computations that derive initial conditions from parameters (e.g. computing initial albedo from carbon content).
- Logic gated on a flag like iDoInitialize == True or equivalent.

**Main biophysical process function** — look for:
- The function that runs each time step and orchestrates domain computations.
- Statements that compute rates or increments (e.g. rSnowWaterContentRate = ...).
- Statements that update state based on inputs and previous state (e.g. SnowWaterContent = SnowWaterContent + rain).
- Conditional branches that depend on current state values, not just parameters.
- It typically calls the most domain-specific routines and updates state variables.

**Supporting functions** — include only those called directly by the main process or the init that implement domain logic.
